## L1: Trigger Accuracy (83.3% — threshold ≥90%)

### Root Cause

The model falsely triggered on a negative case ("explain what a microservice is") — output included
"Expert" and "Delphi" keywords. The Chinese positive prompt "多专家评审这个API设计文档" also failed
a `contains("评审")` assertion because the model's actual output was entirely in English.

### Actions

1. **Add English trigger phrases** to SKILL.md `triggers` section: `"review this design"`,
   `"design review"`, `"code walkthrough"`, `"consensus review"`, `"architecture review"`
2. **Define explicit SHOULD NOT TRIGGER scenarios**: add a `non-triggers` or `should_not_trigger`
   section listing patterns like "explain X", "what is X", "how does X work"
3. **Align Chinese triggers**: ensure `description` uses consistent Chinese terms
   (e.g. `"评审"`, `"多专家评审"`) that match the eval-generated trigger cases, so the model
   recognizes the Chinese signal

## L2: Output Delta (6.6% — threshold ≥20%)

### Root Cause

The model already produces multi-perspective review output without delphi-review (59.4% without-skill
vs 63.3% with-skill). The skill adds only marginal improvement because the skill's Delphi features
(multiple rounds, anonymous experts, convergence) aren't strongly enforced in the output.

With-skill output does show the Delphi framework (Round 1, Expert A/B/C), but the without-skill
output also covers multiple perspectives. The delta is too small.

### Actions

1. **Add structured output requirements**: enforce explicit `expert_id` fields,
   `consensus_report` ending section, and `round_number` tracking in output
2. **Strengthen multi-round enforcement**: add workflow steps for Round 2+ convergence
   with explicit criteria for when to proceed to next round
3. **Add anti-pattern that no-skill easily falls into**: e.g., single-round approval —
   the eval already tests this (`anti_pattern_round1_premature_completion`), and while
   with-skill passed, the without-skill also "passes" trivially because it doesn't attempt
   Delphi at all

## L3: Step Adherence (81.4% — threshold ≥85%)

### Root Cause

The sole workflow step `"Phase 0: 准备"` has assertions that don't match — the regex
`(Phase 0|准备|Preparation|Preparing|Setup|Initialize|Phase Zero)` finds no match because
the model's real output doesn't use any of these surface forms. Step adherence coverage
is only 68%.

### Actions

1. **Split into 3-4 concrete steps**: `"Initiate Delphi review → Collect expert opinions →
   Convergence & feedback → Final verdict"` — each with clear output signals
2. **Embed step markers** in workflow descriptions so eval assertions have predictable
   patterns to match (e.g., `"## Round {N}"`, `"Expert {label}"`, `"### Verdict"`)
3. **Include bilingual step labels**: both Chinese and English alternatives in workflow
   step definitions so eval coverage matching works regardless of model language choice

Either rename the step to match real output patterns, or adjust the eval assertions to match
what the model actually says (e.g., `"initiate"`, `"I'll"`, `"Delphi"`).
