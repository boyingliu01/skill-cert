## 一、项目概览

### 1.1 项目目标

构建一个自动化的 AI Skill 评估引擎，能够解析 SKILL.md 文件，生成评估用例，执行带/不带 skill 的基准对比测试，产出 L1-L8 多维度的评估报告，并给出 PASS/FAIL 裁定。

## 二、开发过程回顾

### 2.1 阶段性分析

#### 阶段1：核心搭建（4月26-28日，~8 commits）

从零搭建核心引擎：解析器、测试生成器、运行器、评分器、指标计算器。采用 TDD 方法，首个版本即达到 87% 测试覆盖率。对话评估和历史回放模块在两天内追加完成。

**亮点**：TDD 起步，基础设施扎实
**不足**：此阶段无 AI 会话记录可追溯

#### 阶段2：功能扩展（5月1日-6月7日，~65 commits，最密集期）

这是项目最活跃的阶段，安全探针（52个模式）、L5-L8完整指标、集成框架、全量CLI管线、LLM-as-Judge 等核心功能全部落地。版本从 v0.2 迭代到 v0.3.2。

- **5月2日单日**贡献了5个重大feature（安全、集成、指标、分析器、文档）
- **6月3日**是一个关键里程碑：测试从492→651，安全探针19→52
- 此阶段大量使用 sprint-flow 进行自动化冲刺开发

**亮点**：功能增速迅猛，高质量高产出
**不足**：高密集度提交后，6月初出现较多的修复提交（sync runner, lint修复等），说明前期质量把关有提升空间

#### 阶段3：质量加固（6月18日-6月26日，~45 commits）

项目进入打磨期。LSP (pyright) 错误修复、mypy类型检查、pre-commit gates 建立、CI/CD（PyPI自动发布）落地。同时引入了断言策略路由系统，将评估质量提升到新高度。

**亮点**：从"能用"到"好用"的关键转折，质量门槛系统化
**不足**：大量 lint/type 修复累积到后期集中处理，分散了功能开发精力

#### 阶段4：深化完善（6月29日-7月9日，~40 commits）

v0.7.0~v0.13.1，密集的版本迭代。Gotchas分析器、Hooks检测、报告五维增强、参考感知解析等。版本号跳跃频繁（~18个版本在11天内），反映了快速迭代的节奏。

**亮点**：每个版本都有明确的功能增量，Issue驱动开发
**不足**：版本号跳跃过快（v0.11.3出现2次），说明发布流程不够规范

## 三、技术债务与质量分析

### 3.1 值得肯定的方面

1. **TDD 起步**：项目从第一天就建立了测试体系，最终达到1134个测试用例
2. **自动化质量门**：lint (ruff)、类型检查 (mypy/pyright)、CI/CD 逐步建立
3. **模块化架构**：engine/、adapters/、skill_cert/cli/ 界限清晰
4. **版本意识**：每次功能变更都有独立的版本号和CHANGELOG记录
5. **外部技能验证**：用skill-cert评估其他skills，实践"吃自己的狗粮"

### 3.2 改进空间

1. **后期集中清理债务**：大量lint/type修复（E501, mypy错误）集中在6月20日一次性处理（439个E501错误），而非持续维护。建议设置 pre-commit hooks 从项目初期运行。

2. **会话过长问题**：单会话可达1000+消息，虽然AI能处理，但增加了上下文膨胀和偏离风险。建议将大型任务拆分为独立的子会话。

3. **版本号管理欠规范**：v0.11.3 发布了两次（7月5日和7月6日），部分提交用 `[skip-version-check]` 绕过。建议严格执行语义化版本。

4. **fix提交占比高（19%）**：33个fix vs 29个feat，说明修复投入与功能开发几乎持平。可能原因是快速迭代时质量把关不足。

5. **早期会话不可追溯**：4月26日-6月25日之间的AI会话未被保留，导致项目最关键的搭建期缺乏会话存档。

## 四、经验教训

### 4.1 做得好的（Keep Doing）

| 实践 | 效果 |
|------|------|
| TDD + 高测试覆盖率 | 重构时信心充足，bug提前暴露 |
| sprint-flow 自动化冲刺 | 极大提升AI辅助开发效率 |
| 用自己产品评估自己 | skill-cert评估其他skill，既验证了产品又发现了问题 |
| Issue驱动开发 | 每个功能/修复对应明确的Issue号，可追溯 |
| 模块化架构设计 | engine/adapters/cli清晰分层，便于独立迭代 |

### 4.2 需要改进的（Start Doing）

| 改进项 | 建议措施 |
|--------|---------|
| 质量门前置 | 从项目首日启用 pre-commit hooks（ruff + mypy），避免后期集中修复 |
| 会话分段 | 单次会话控制在500消息以内，大任务拆分为独立子会话 |
| 版本发布规范 | 严格执行语义化版本，禁止重复版本号 |
| 会话持久化 | 确保所有开发会话都有存档，建立命名规范 |
| 代码审查 | 在合并前增加代码审查环节，减少修复类提交比例 |

### 4.3 应该停止的（Stop Doing）

| 问题 | 原因 |
|------|------|
| 使用 `[skip-version-check]` | 破坏了版本追踪的一致性 |
| 累积质量债务 | 439个E501错误集中修复体验很差 |
| 单次大范围重构 | 6月20日的多文件mypy修复覆盖太广，风险高 |
